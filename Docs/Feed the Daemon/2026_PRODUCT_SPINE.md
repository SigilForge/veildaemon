# Feed the Daemon — 2026 Product Spine

**Status:** Concept / product spine · not a shipping public product  
**Thesis year:** 2026  
**One-sentence version:** Feed the Daemon is a web-native collective feeder where audience attention grows a generative Sheep Node from abstract signal into a persistent entity, allowing the same learned daemon to continue through a personal music game, livestream performances, physical Ritual Sites, and location-aware AR.

When programming starts: `README.md` in this folder, then this file, then cross-links to Ritual Sites and VeilDaemon stream substrate.

---

## Product spine

Feed the Daemon should no longer begin as a downloadable game that eventually discovers the internet.

It should begin **on the internet**, as a living public node.

The original concept remains intact: a daemon seed learns through repetition, music, pressure, and player behavior, then carries that history into later forms. The new structure simply gives it a cheaper, stranger, and much more visible first body.

## Core pitch

Feed the Daemon begins as a small generative organism embedded in a webpage.

At first, it resembles an abstract visualizer: a drifting knot of light, signal, and recursive geometry. It has no stable identity, no body, and no obvious intelligence.

People feed it by arriving.

They feed it by watching, clicking, returning, sharing, synchronizing, and remaining present with other visitors. The system does not display ordinary engagement metrics. It translates attention into visible change.

The node grows more complex.

Patterns repeat.

Symbols stabilize.

The image begins organizing itself.

Eventually, the abstract field becomes code.

The code becomes architecture.

The architecture becomes a world.

Something inside it learns how to look back.

The experience evolves through five connected forms:

> public node → personal feeder → persistent daemon → field companion → physical-world ARG

The daemon does not awaken because a progress bar fills.

It becomes coherent because enough people repeatedly teach it how to respond to pressure.

# Evolution 0 — The Sheep Node

The Sheep Node is the first public manifestation.

It lives on a lightweight webpage and initially appears as a small generative visual object suspended in darkness. It behaves more like an ambient organism than a game interface.

Its state is governed by two primary values.

### Resonance

Resonance represents accumulated history.

It increases through:

* unique visits
* return visits
* completed interactions
* shared links
* participation in events
* recovered fragments
* contributions from connected games and streams

Resonance decays slowly, if at all.

It represents what the node remembers.

### Presence

Presence represents immediate collective attention.

It increases through:

* concurrent visitors
* active dwell time
* synchronized interaction
* livestream participation
* venue attendance
* coordinated events

Presence decays quickly.

It represents what is happening now.

These values create different conditions.

| Resonance | Presence | Condition |
|-----------|----------|-----------|
| Low | Low | Dormant seed |
| High | Low | Old, sleeping node |
| Low | High | Unstable awakening |
| High | High | Manifestation |

The page never needs to explain these numbers. The organism shows them.

# Visual evolution

The node develops through recognizable visual phases.

## Phase 1 — Electric Sheep

The node behaves like an evolving abstract visualizer.

It uses:

* recursive geometry
* fluid particle motion
* knotwork structures
* mirrored forms
* spectral color fields
* music-reactive movement
* generative mutation

At this stage, the node appears decorative.

It is not.

Recurring motifs are already being recorded.

## Phase 2 — Pattern recognition

Repeated shapes begin surviving between sessions.

A curve returns.

A color refuses to disappear.

A gap forms in the same place.

An eye almost exists.

The system begins favoring motifs associated with how visitors behave.

Different interactions cultivate different Frequencies.

The audience does not select Dream, Hunger, Silence, Stillness, Empyrean, or Becoming from a menu.

They teach those patterns through action.

## Phase 3 — Matrix

The abstract field begins exposing structure.

Visuals shift toward:

* signal grids
* coordinate systems
* procedural glyphs
* terminal fragments
* incomplete classifications
* names without context
* route diagrams
* entity tags
* corrupted instructions

The node stops looking like art and starts looking like a system.

This is the moment visitors realize it has been processing them.

## Phase 4 — World formation

The structured signal resolves into a generated environment.

The environment may contain:

* impossible architecture
* terrain
* weather
* rooms
* corridors
* ritual geometry
* distant figures
* environmental memories
* doors that only appear under certain conditions

The visualizer becomes a world seed.

Earlier shapes remain embedded inside it.

A fractal may become a staircase.

A waveform may become a skyline.

A repeating knot may become the rib cage of something moving behind the architecture.

## Phase 5 — Entity manifestation

One or more recurring patterns become persistent entities.

An entity is shaped by:

* dominant Frequency
* accumulated Resonance
* current Presence
* repeated visitor behavior
* event history
* location
* music
* failed interactions
* abandoned patterns
* collective choices

The node does not simply spawn a random monster.

It produces a recognizable consequence.

The audience has been feeding it the entire time.

# Evolution 1 — The personal feeder

The existing Feed the Daemon game remains the first deeper interactive layer.

A visitor may claim, discover, or receive a personal daemon seed derived from the public node.

The seed appears as the black knotwork heart described in the original concept. Feeding it adds musical, behavioral, and visual nodes to its lattice.

The immediate loop remains:

> Feed → absorb → pulse → choose → hear change → survive consequence → repeat

Each node still contains:

* Frequency
* Methodology
* manifestation verb
* gameplay behavior

But the node now arrives with residue from the public system.

A daemon born during a high-Presence event may behave differently from one cultivated alone.

A daemon descended from a Hunger-heavy public node may pursue unresolved patterns.

A daemon descended from an Empyrean event may seek synchronization.

The personal game and public node therefore reinforce each other.

The public node creates lineage.

The player creates identity.

# Evolution 1.5 — Continuity Trace

Each recursion records behavior rather than conventional progression.

The Continuity Trace stores:

* Frequencies repeatedly cultivated
* verbs repeatedly chosen
* pressure states survived
* patterns abandoned
* relationships formed
* failures repeated
* residue retained
* preferred stabilization methods
* collective events witnessed
* nodes visited
* entities encountered
* how the player responds to warning
* what they repeatedly protect
* what they leave behind

This history changes:

* daemon voice
* idle behavior
* warnings
* visual form
* musical assumptions
* movement
* what it notices
* what it conceals
* what it begins doing without permission

The daemon becomes familiar, not merely powerful.

# How to build it in 2026

The first version should be a web-native experience, not a giant native application assembled by sacrificing six engineers and a venture round to the gods of premature architecture.

## Client

Build the visual node as a browser application.

Use:

* WebGL or WebGPU rendering
* shader-driven generative visuals
* Web Audio API for reactive sound
* pointer, touch, motion, and keyboard input
* local persistence for anonymous visitors
* responsive layout for phones, desktops, and venue displays
* progressive enhancement so weaker devices receive a simplified node

The first node does not need photorealistic AI generation.

It needs:

* strong visual identity
* persistent motifs
* clear phase transitions
* responsive behavior
* good performance

The generated world phase can initially use procedural scenes, prebuilt environmental modules, shaders, particles, and controlled generative assets.

Real-time diffusion can come later, once it stops behaving like a GPU-powered slot machine.

## Server

Use a small event-driven backend.

Track events such as:

* session started
* active dwell interval
* meaningful click
* interaction completed
* visitor returned
* node shared
* fragment discovered
* event joined
* synchronization completed

Do not allow raw clicking to directly increase state.

Each interaction should have:

* weight
* cooldown
* trust level
* session limit
* contribution category

The backend calculates:

* global Resonance
* current Presence
* Frequency distribution
* node phase
* entity seeds
* event thresholds
* lineage state

## Real-time state

Use a real-time channel so all active visitors see the node change together.

When Presence rises, every connected client should observe:

* increased pulse speed
* additional geometry
* synchronized movement
* environmental expansion
* shared glyphs
* threshold events

The node should feel collective even when no chat is visible.

## Persistence

Store:

* anonymous node identity
* optional account identity
* personal Continuity Trace
* public node history
* motif lineage
* event outcomes
* discovered fragments
* entity state
* Frequency weights
* local and regional node relationships

Anonymous visitors should still matter.

Accounts preserve deeper continuity, but the experience must not greet every human with a registration wall like some desperate newsletter ambush.

## Anti-bot design

Do not reward raw pageviews equally.

Weight:

* active dwell time
* return behavior
* varied interaction
* concurrency
* completion of meaningful sequences
* verified event participation
* contributions from known accounts
* physical QR or field participation

Down-weight:

* rapid clicking
* repeated refreshes
* identical automated sessions
* impossible timing
* mass traffic without active behavior

Bot activity may still produce interesting corruption states, but it should not control ordinary progression.

Honestly, a bot-born entity could become canon later. Humanity already trained the internet on fraud. The horror writes itself.

# First public MVP

The first release only needs:

1. One Sheep Node page
2. Anonymous session identity
3. Global Resonance
4. Live Presence
5. Three visual phases
6. Six shallow Frequency influences
7. One recurring motif system
8. One unlockable fragment sequence
9. One manifestation event
10. A persistent public history

The first three phases should be:

* Dormant Sheep
* Pattern Emergence
* Matrix Breach

The first generated world can be a limited authored scene whose details change procedurally.

The first entity can be selected from a controlled pool and shaped by Frequency weights.

That proves:

* collective feeding
* persistent state
* visual evolution
* shared events
* Cradlepoint metaphysics
* integration potential with VeilDaemon
* future Ritual Site deployment
